## **1\. 프로젝트 개요 (Project Overview)**

## **1.1 배경 및 문제 정의**

- **정보 탐색 및 분석의 과부하:** 부동산 시장의 방대한 공공데이터와 비정형 뉴스 데이터를 개인이 통합 분석하여 의사결정하기에는 물리적 한계가 존재함.
- **사용자 액션의 단절:** 기존 플랫폼은 단순 정보 조회에 그쳐, 실제 방문 예약이나 중개인과의 협상 등 실질적인 거래 프로세스로 이어지는 비즈니스 로직이 부족함.
- **공급자-수요자 간 미스매칭:** 매도자의 급매 요구사항과 중개사의 전문성이 효율적으로 연결되지 않아 발생하는 거래 지연 문제 해결이 필요함.

## **1.2 기획 목표**

- **LLM 기반 인터페이스 혁신:** 자연어 처리를 통해 사용자의 복잡한 요구사항을 정교한 검색 필터로 자동 전환.
- **트랜잭션 중심의 서비스 설계:** 단순 조회를 넘어 예약, 입찰, 상태 관리 등 소프트웨어 공학적 복잡도가 높은 유스케이스 구현.
- **다중 이해관계자 매칭:** 매수자, 매도자, 중개인 간의 3-Way 상호작용이 가능한 통합 플랫폼 구축.

## **2\. 시스템 아키텍처 및 액터 설계 (System Architecture & Actors)**

## **2.1 분산 시스템 구조 (Multi-tier Architecture)**

**본 시스템은 사용자 접근성 및 데이터 처리 효율을 극대화하기 위해 Multi-tier 분산 구조를 채택함.**

- **Presentation Tier:** Streamlit 기반의 웹 인터페이스로 사용자의 요청을 수집하고 분석 결과를 시각화함.
- **Application/Logic Tier:** FastAPI 서버가 비즈니스 로직(예약 확정, 입찰 매칭) 및 LLM 기반 자연어 파싱을 처리함.
- **Data Tier:** 실거래가 정형 데이터(PostgreSQL)와 뉴스 비정형 데이터(Vector DB)를 분리 저장하여 고속 검색을 지원함.

## **2.2 시스템 액터 (System Actors)**

- **매수자 (Buyer):** 검색 또는 자연어를 통해 매물을 검색 및 추천받고 방문 예약을 수행하는 주체.
- **매도자 (Seller):** 매물 정보를 등록하고 중개사를 대상으로 수수료 역경매 입찰을 생성하는 주체.
- **중개사 (Agent):** 매수자의 예약을 관리하고 매도자의 입찰에 참여하여 거래를 성사시키는 주체.
- **외부 데이터 API (External System):** 국토교통부 실거래가 정보 및 포털 뉴스 데이터를 실시간으로 제공하는 연동 시스템.

## **3\. 서비스 기능 및 유스케이스 (Service Features & Use Cases)**

## **3.1 핵심 기능 명세**

- **지도 기반 예약 및 방문 지원:** 추천 매물의 위치를 지도에서 확인하고 중개인과 실시간 방문 일정을 확정하는 스케줄링 시스템.
- **AI 지능형 필터링:** "역세권 학원가 근처 10억대 매물"과 같은 자연어 입력을 LLM이 분석하여 검색 조건으로 자동 변환.
- **매도인-중개인 역경매 입찰:** 매도인이 매물과 요구사항을 올리면, 중개사들이 수수료와 조건을 입찰하여 매칭되는 시스템.
- **리스크 및 수익률 분석:** 실거래가 기반의 리스크 지수 시각화 및 과거 데이터 기반 수익률 시뮬레이션 제공.

## **3.2 유스케이스 및 대안 흐름 (Alternative Flows)**

**\[UC-01\] 방문 예약 및 사후 관리 시스템:**

- **기본 흐름:** 사용자가 매물을 예약하고 방문(임장)을 완료하면, 시스템은 자동으로 **'후기 작성' 알림**을 발송함.
- **후기 기능:** 사용자는 중개사의 친절도, 매물의 실제 상태(사진과의 일치 여부) 등을 별점과 텍스트로 기록하며, 이는 해당 매물 및 중개사 평판에 즉각 반영됨.

**\[UC-02\] 자연어 매물 검색:** 입력어가 모호하거나 조건에 부합하는 매물이 없을 경우, 유사 지역을 제안하거나 검색 범위 확대를 권고함.

## **4\. 상세 설계 및 트랜잭션 모델 (Detailed Design)**

## **4.1 트랜잭션 상태 전이 모델 (State Transition Model)**

**예약 및 거래 생애주기:**

- **Pending(대기)** → **Confirmed(확정)** → **Visited(방문 완료)** → **Reviewed(후기 완료)**
- **동시성 제어:** 여러 사용자가 동시에 동일 매물의 '후기 작성'을 시도하더라도, 각 예약 번호(ID)별로 독립적인 트랜잭션을 보장하여 데이터 꼬임을 방지함.

**역경매 입찰 상태 생애주기:**

- **입찰 대기** → **입찰 진행 중** → **중개사 낙찰** → **거래 완료/취소**
- **동시성 제어(입찰):** 여러 중개사가 동시에 '낙찰' 버튼을 누르더라도, 시스템은 순차 처리를 적용함.
- 가장 먼저 도착한 요청 외의 나머지 동시 요청은 자동으로 차단하여 **중복 낙찰 사고를 원천 방지**함.

## **4.2 기술 스택 (Tech Stack)**

- **Language/Web:** Python 3.x, Streamlit, FastAPI.
- **AI/Data:** OpenAI API, LangChain, Pandas, Plotly.
- **Database:** PostgreSQL (Relational), Pinecone (Vector).

**L0_Prompt : Problem Descrption**

**\[역할 지정\]**

- Java로 AI 기반 부동산 매칭 및 거래 지원 플랫폼인 \*\*'집피티(ZIP-PT)'\*\*를 구현하라.
- 외부 프레임워크(Spring 등) 없이 \*\*순수 Java(Pure Java)\*\*로 작성한다.

**\[컨텍스트\]**

- 제공된 집피티(ZIP-PT)의 서비스 개념과 주요 기능을 기반으로 구현한다.
- 주요 액터: 매수자(Buyer), 매도자(Seller), 중개사(Agent).

**\[생성 지시\]** 다음 핵심 비즈니스 로직을 포함하여 구현하라:

- **역경매 입찰 및 낙찰:** 매도인이 매물을 등록하면 중개사들이 수수료 조건을 입찰하고, 최종 낙찰자가 선정되는 프로세스.
- **매물 검색 및 방문 예약:** 매수자가 매물을 조회하고, 중개사의 일정에 맞춰 방문 예약을 신청하는 기능.
- **상태 관리:** 예약의 생애주기(Pending → Confirmed → Visited → Reviewed)를 관리.
- **후기 작성:** 방문 완료(Visited)가 확인된 건에 한해 별점과 텍스트 후기를 등록하는 기능.

**\[제약 조건\]**

- **예약 중복 방지:** 중개사의 기존 일정과 겹치는 시간대의 예약은 거부한다.
- **입찰 동시성 제어:** 여러 중개사가 동시에 낙찰을 시도할 경우, 서버 수신 순서에 따른 순차 처리를 적용하여 중복 낙찰을 차단한다.
- **후기 권한 제어:** 방문 완료 상태가 아닌 예약 건에 대한 후기 작성 시도는 거부한다.
- **출력 방식:** 별도의 UI 없이 콘솔 출력(System.out)으로 시스템 흐름과 처리 결과를 표시한다.
